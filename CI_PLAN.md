# CI Plan — GitHub Actions for soloiv (deferred)

Status: **deferred.** The test suite is green locally (20/20, serial and
parallel) and now runs entirely off the committed `tests/fixtures/` tree, so
it no longer depends on the large local `test_data/`. The remaining work to
run it on GitHub CI is captured here for a future session.

## What's already done (prerequisites)

- **Fixtures are in-repo.** `tests/fixtures/{ground,airborne,ship}/{dorade,cf1,cf2}`
  hold small (range-clipped) real sweeps, ~12 MB total, regenerated by
  `tests/fixtures/make_fixtures.sh`. No data download needed in CI.
- **Stable fixture discovery.** Tests resolve data via CTest `WORKING_DIRECTORY`
  and the `SOLOIV_FIXTURE_DIR` compile define (env-overridable). No dependence
  on absolute local paths.
- **Suite is deterministic.** No flaky/crashing tests; the earlier failures
  were all path-drift (data moved out of `test_data/` root), now fixed.

## The main blocker: LROSE on Linux CI

soloiv links the LROSE `libRadx`/`libNcxx` (currently `/Users/mmbell/lrose`).
CI needs LROSE present on the runner. Options, roughly in order of preference:

1. **Prebuilt LROSE container image.** Use an `lrose-core` Docker image (or a
   thin image that `apt`/`conda` installs a published LROSE build) as the CI
   job container. Fastest, most reproducible. Verify the image ships the Radx
   headers + static/shared libs soloiv's CMake `find_package(Radx)` expects.
2. **Install a published LROSE release** (tarball/conda) in a workflow step.
   Avoids maintaining an image; adds per-run install time.
3. **Build LROSE from source in CI** (cache the build). Most flexible, slowest;
   only if 1–2 don't fit.

`CMakeCache` currently points `RADX_INCLUDE_DIR`/`RADX_LIBRARY` at the local
install; CI must set these (or `CMAKE_PREFIX_PATH`) to wherever LROSE lands.

## Headless display for the GTK4 GUI tests

Several tests open real GTK4 windows (examine/edit/param/swpfi/multi-frame,
plus the data-path tests that start the app). On Linux runners these need a
display:

- Run the suite under **Xvfb** (`xvfb-run -a ctest ...`), or set up a virtual
  framebuffer service. macOS runners have a native display but cost more CI
  minutes.
- GTK4 also needs its runtime deps (`libgtk-4`, cairo, pango) installed on the
  runner.

Optional refinement: split tests into `data` and `gui` CTest labels so a fast
`data`-only job can gate without a display, with `gui` under Xvfb.

## Known issue to file: legacy DORADE byte parser vs Radx-written DORADE

The in-tree DORADE byte parser (`translate/swp_file_acc.c`) rejects
Radx-written DORADE as "Current sweep file is garbage!" — it trips the block
`sizeof_struct` range check (`dd_defines.h: MAX_REC_SIZE`) somewhere in the
block loop. Genuine archival DORADE is little-endian, `SSWB`-first; Radx (and
soloiv's own Editor save, which goes through Radx) writes big-endian,
`COMM`-first sweeps. The parser's byte-swap path handles the first block but
still fails later.

Impact: the legacy parser can only read original archival DORADE, not anything
the modern stack writes. This is why no trimming tool can produce a small
legacy-readable DORADE, and why `test_radx_airborne` was switched to read all
formats through Radx. Since DORADE is being phased out (pipe old data through
`RadxConvert` first), this is low priority — but pinning the exact offending
block would let the legacy parser read modern DORADE if ever needed. Fits the
DORADE-robustness bug class.
